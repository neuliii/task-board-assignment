# 설계 결정 (DECISIONS)

> 이 문서는 평가에서 **20%**를 차지합니다. "무엇을 했는지"보다 **"왜 그렇게 했는지"**를 봅니다.
> 아래 항목을 채워 주세요. 코드로 답이 되는 부분은 파일 경로를 함께 적어도 좋습니다.

## 1. 상태 구조
### 상태 관리
애플리케이션의 상태 관리는 Zustand를 사용했습니다.
Task 목록, 로딩 상태, 에러 상태, 낙관적 업데이트에 필요한 상태를 하나의 Store에서 관리하도록 구성했습니다.

### 선택 이유
* React Context보다 구현이 단순하고 Boilerplate가 적었습니다.
* Redux Toolkit보다 설정 비용이 낮아 과제 규모에 적합하다고 판단했습니다.
* 비동기 액션과 상태를 하나의 Store에서 관리하여 코드의 응집도를 높일 수 있었습니다.

### 고려했던 대안
React의 local state만 사용하는 방법도 고려했지만, CRUD와 낙관적 업데이트, Rollback, Race Condition 처리가 추가되면서 여러 컴포넌트에서 동일한 상태를 공유해야 했기 때문에 전역 상태 관리가 더 적합하다고 판단했습니다.

## 2. 낙관적 업데이트 & 롤백
모든 생성(Create), 수정(Update), 삭제(Delete), 이동(Move)은 먼저 UI를 변경한 뒤 서버 요청을 수행하도록 구현했습니다.
요청이 성공하면 그대로 유지하고, 실패하면 이전 상태를 복원한 뒤 사용자에게 오류 메시지를 표시합니다.

### 롤백 방식
롤백은 서버 재조회가 아닌 이전 스냅샷 복원 방식을 선택했습니다.

### 선택 이유
* 즉시 이전 상태를 복원할 수 있어 사용자 경험이 자연스럽습니다.
* 불필요한 네트워크 요청을 줄일 수 있습니다.
* 과제에서 요구한 “실패 시 정확한 Rollback”을 가장 단순하게 만족할 수 있었습니다.

## 3. 경쟁 상태(race condition)
같은 Task를 빠르게 여러 번 이동하는 상황을 고려하여 요청마다 고유한 순번(moveId)을 관리했습니다.
응답이 도착했을 때 현재 저장된 최신 요청과 비교하여 오래된 요청이면 결과를 무시하도록 구현했습니다.

### 선택 이유
네트워크 지연으로 인해 이전 요청이 나중에 도착하는 경우가 발생할 수 있으므로, 오래된 응답이 최신 상태를 덮어쓰지 않도록 하기 위함입니다.

## 4. 대량 데이터 성능 (5,000개)
5,000개의 Task를 모두 렌더링하지 않도록 @tanstack/react-virtual을 사용하여 리스트를 가상화했습니다.
화면에 보이는 항목만 렌더링하여 DOM 노드 수를 크게 줄였고, 스크롤과 드래그 성능을 개선했습니다.

### 선택 이유
* 대량 데이터에서도 안정적인 성능을 제공합니다.
* React 생태계에서 활발히 유지보수되고 있습니다.
* 필요한 기능만 적용하기 쉬웠습니다.

### 포기한 부분
추가적인 메모이제이션이나 세부 렌더링 최적화는 현재 성능으로 요구사항을 만족한다고 판단하여 적용하지 않았습니다.

## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)

### 네트워크 완전 단절
현재는 요청 실패 시 Rollback을 수행하고 오류 메시지를 표시하도록 구현했습니다.
오프라인 상태를 별도로 감지하거나 오프라인 큐를 구현하지는 않았습니다.

### 409 충돌
Priority 2 범위로 판단하여 별도의 UX는 구현하지 않았습니다.
현재는 서버 오류와 동일하게 실패를 사용자에게 알리고 Rollback하도록 처리했습니다.

### 실패한 쓰기의 재시도
자동 재시도는 구현하지 않았습니다.
자동 재시도는 중복 요청이나 의도하지 않은 상태 변경을 발생시킬 수 있다고 판단했으며, 제한된 개발 시간에서는 핵심 요구사항(Priority 1)의 안정성을 우선했습니다.

## 6. 그 외 트레이드오프 / 남은 한계
Priority 1 요구사항을 우선 구현하기 위해 Priority 2 기능은 구현하지 않았습니다.

### 현재 구현의 한계는 다음과 같습니다.
* 409 충돌에 대한 별도 사용자 경험은 제공하지 않습니다.
* 자동 재시도 및 Backoff는 구현하지 않았습니다.
* 다중 탭 동기화는 지원하지 않습니다.
* 키보드 접근성과 검색 디바운싱, 다중 필터는 구현하지 않았습니다.

제한된 개발 시간에서는 핵심 요구사항인 낙관적 업데이트, Rollback, Race Condition 처리, CRUD, 대량 데이터 렌더링 최적화 및 테스트의 완성도를 높이는 것을 우선순위로 두었습니다.
